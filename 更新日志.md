## [2026-05-20] Bug修复：BloomFilter未加载用户ID + FeedServiceImpl编译错误 + 单元测试修复

### 问题背景

1. **BloomFilter未加载用户ID**：ContainerInitializerListener在周期性重建布隆过滤器时，只加载了videoDao.findAllIds()和postDao.findAllIds()，没有加载userDao.findAllIds()。导致UserServiceImpl的三层缓存防护（布隆过滤器→Redis缓存→分布式锁+双重检查）在第一层就拦截了所有合法用户ID，用户查询永远返回"用户不存在"。
2. **FeedServiceImpl编译错误**：lambda表达式中引用了非final变量`minScore`和`limit`，Java要求lambda捕获的局部变量必须是final或effectively final，导致编译失败。
3. **单元测试错误**：VideoServiceImplTest和PostServiceImplTest存在5个测试错误，主要原因是BloomFilter未初始化导致缓存防护误判、Redis缓存污染导致测试间数据干扰、以及Mock stub未被调用时抛出UnnecessaryStubbingException。

### 修复内容

**Bug1 - BloomFilter未加载用户ID**：
- ContainerInitializerListener.java：在布隆过滤器周期性重建方法中，增加`userDao.findAllIds()`的加载，将所有正常状态的用户ID也加入布隆过滤器
- 修复后，布隆过滤器同时包含视频ID、动态ID和用户ID，三层缓存防护对用户查询正常生效

**Bug2 - FeedServiceImpl lambda final变量修复**：
- FeedServiceImpl.java：在lambda表达式外创建`final double finalMinScore = minScore;`和`final int finalLimit = limit;`两个final副本变量
- lambda表达式内部改为引用finalMinScore和finalLimit，满足Java编译要求

**Bug3 - 单元测试修复（5个错误）**：
- VideoServiceImplTest：新增`@BeforeEach setUp()`方法，在每次测试前将测试用视频ID加入BloomFilter，并清理Redis缓存，防止缓存防护误判和缓存污染
- PostServiceImplTest.GetDetailTests：新增`@BeforeEach cleanCache()`方法，在每次测试前清理Redis缓存，防止getDetail_deleted测试写入的空值标记(null marker)污染后续测试
- 使用`lenient().when()`替代`when()`，对可能因Redis缓存命中而未被调用的Mock stub放宽校验，避免UnnecessaryStubbingException

## [2026-05-19] 消息通知集成：关注/点赞/评论自动发送通知

### 背景

为了让用户在收到关注、点赞、评论时能及时收到消息通知，在 FollowServiceImpl、LikeServiceImpl、CommentServiceImpl 三个核心服务中集成了通知发送逻辑。通知发送采用 try-catch 包裹，确保通知发送失败不影响主业务流程。

### 改动

**1. FollowServiceImpl.java** — 关注成功后通知被关注者：
- 新增 `@Autowired NotificationServiceImpl notificationService` 依赖注入
- 在 `toggle()` 方法中，关注成功后构建 FOLLOW 类型通知并发送
- 通知内容示例：`用户【xxx】关注了你`

**2. LikeServiceImpl.java** — 点赞成功后通知内容拥有者：
- 新增 `@Autowired NotificationServiceImpl notificationService` 依赖注入
- 新增 `@Autowired UserDao userDao` 依赖注入（用于查询点赞者昵称）
- 新增 `Comment`、`Post`、`Notification`、`User` 实体类 import
- 在 `toggle()` 方法中，点赞成功后根据 targetType 查找内容拥有者，构建 LIKE 类型通知并发送
- 自己赞自己的内容不发通知（`!targetOwnerId.equals(userId)` 判断）

**3. CommentServiceImpl.java** — 评论成功后通知内容拥有者：
- 新增 `@Autowired NotificationServiceImpl notificationService` 依赖注入
- 新增 `@Autowired VideoDao videoDao` 依赖注入（用于查询视频拥有者）
- 新增 `Notification`、`Post`、`Video` 实体类 import
- 在 `create()` 方法中，评论成功后根据 targetType 查找内容拥有者，构建 COMMENT 类型通知并发送
- 自己评论自己的内容不发通知

### 设计亮点

- **通知与主流程解耦**：通知发送用 try-catch 包裹，失败只记 warn 日志，不影响关注/点赞/评论主业务
- **防自通知**：点赞和评论场景下，自己操作自己的内容不会产生通知
- **通知类型区分**：FOLLOW / LIKE / COMMENT 三种类型，便于前端分类展示

## [2026-05-12] 数据库优化：物理外键改为逻辑外键

### 背景

本项目数据库原使用了物理外键（FOREIGN KEY + ON DELETE CASCADE），在大型互联网高并发场景下存在性能瓶颈，且与软删除策略冲突。

### 改动

**`backend/init.sql`** — 移除所有表的物理外键约束，但保留索引：

移除的外键清单：

| 表名 | 原外键字段 | 引用表 |
|------|-----------|--------|
| `video` | `user_id` | `user(id)` |
| `comment` | `user_id`、`parent_id` | `user(id)`、`comment(id)` |
| `follow` | `follower_id`、`followee_id` | `user(id)` |
| `like_record` | `user_id` | `user(id)` |
| `favorite_folder` | `user_id` | `user(id)` |
| `favorite_record` | `folder_id`、`user_id` | `favorite_folder(id)`、`user(id)` |
| `post` | `user_id` | `user(id)` |
| `role_permission` | `role_id`、`permission_id` | `role(id)`、`permission(id)` |
| `user_role` | `user_id`、`role_id` | `user(id)`、`role(id)` |

保留的索引（对查询性能至关重要）：
- 所有 `idx_*` 索引
- 所有 `UNIQUE KEY` 约束

### 优势

1. **高性能**：消除高并发写入时的外键锁竞争
2. **兼容软删除**：项目使用 `status` 字段做软删除，与物理外键的硬删除策略不冲突
3. **扩展性好**：分库分表、数据迁移更灵活
4. **应用层控制**：数据一致性由业务逻辑保证（如删除视频时校验作者身份）

- **问题**：前端发送 `username` 字段，后端 `RegisterDTO` 期望 `nickname`，导致 Jackson 反序列化失败，返回 50000 系统内部异常

  - `frontend/src/components/NavBar.vue`：`userStore.username` → `userStore.nickname`

- **问题**：`AuthInterceptor` 从 Redis 反序列化 User 对象时，`ObjectMapper` 未注册 `JavaTimeModule`，导致 `LocalDateTime` 类型的 `createdAt` 字段无法解析，抛出异常后被 catch 为"未登录"

  - `backend/.../controller/UserController.java`：同上
  - `backend/.../filter/GlobalExceptionFilter.java`：同上（预防性修复）

### 修复头像上传失败（40000 请选择要上传的文件）

- **问题**：前端 `FormData` 字段名为 `avatar`，后端 `request.getPart("file")` 期望字段名为 `file`
- **修改文件**：
  - `frontend/src/api/user.js`：`formData.append('avatar', file)` → `formData.append('file', file)`
